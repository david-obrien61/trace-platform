# Front-Door Arc Recon — builtwithCAI register → invite → scrape-while-away → prepared reveal

**Date:** 2026-06-26 · **Author:** Thunder · **Type:** READ-ONLY recon (no code, no schema, no `[TRACE:*]` touched) · **Bar:** Recon COMPLETE (doc-only)

**Why this recon:** We stop patching Cultivar onboarding bug-by-bug. The real target is the builtwithCAI **front-door arc**, which subsumes most of today's individual bugs (address completeness, address-conflict-not-raised, "review my inventory" over-promise, wall-of-fields vs progressive screens, the empty dashboard). This doc produces the TRUE MAP — what the docs say, what the code does, where the gaps are — so the arc can be **promoted once** instead of patched in pieces. And it answers the one landmine (identity/auth coupling) before any build.

**The arc as the owner describes it:** user registers via ONE small screen → system sends an **INVITE** (return link) → the invite buys **TIME** while the system **scrapes/interrogates the business DEEPLY in the background** → user **RETURNS** via the invite to a reveal that is **already prepared** ("here's what we found," deep, pre-populated). Plus two load-bearing sub-pieces: a **structured-query architecture** (the vertical is the entry point; it assembles `user input + vertical context` into a structured query handed to a shared, vertical-agnostic `discovery_engine`) and a **validation branch** (URL → query the site; address-but-no-URL → run against Google; raise conflicts between entered vs. source).

### D2 — Smallest coherent way to PROMOTE THE WHOLE ARC ONCE

**The honest finding: the full async arc cannot promote clean — its first move (scrape before account) is the exact inversion of how the code is wired today (account before scrape). See Part C.** So there are two real sizes:

- **Smallest coherent promote that does NOT touch auth (recommended first):** keep the **synchronous** reveal that already works (`DiscoveryGlimpse` is BUILT and runs as a signup vertical step), and close the three subsumed bugs around it *without* inverting the account/scrape order:
  1. **Wire `compareEnteredVsSite` into `DiscoveryGlimpse`** (it's built, zero callers) + **add `address` to the extraction schema and the compare field set** → the recognition/address-conflict moment fires. (Reuse + thin net-new; no auth.)
  2. **Wire `populateCatalog` into the post-create discovery path** → kills the empty dashboard / "review my inventory" over-promise. (Reuse the CLI's function; no auth.)
  3. **Add a Google/address validation branch** behind a geocoder key → the address-but-no-URL fork. (Net-new; needs a key; **hard-gated on a geocoder** per `2026-06-25-routing-seeder-seam-recon.md` — independent of auth.)
  This is **mostly reuse**: the engine, the reveal, the compare function, and `populateCatalog` all exist. Net-new is the address field, three wirings, and the geocoder. **None of it touches `OwnerSignup.tsx`'s `signUp` call, the businesses-insert order, or `acceptInvitation.ts`.**

- **The full async, invite-gated, scrape-while-away arc — FORCES the auth reconciliation** (slow, owner-decision). It requires running discovery/scrape **before** account creation and a **pre-auth prospect token**, which today does not exist (the only token, `acceptInvitation`, *creates* the auth user). That inversion touches Off-Limits-adjacent surfaces: the `signUp` call site (`OwnerSignup.tsx:397`), the businesses-insert-needs-`owner_id` ordering (`OwnerSignup.tsx:254,282`), and a new pre-auth→`owner_id` **claim/merge** when the prospect eventually signs up. **This is net-new identity plumbing, not a UI promote.**

**Reuse vs net-new, named:**
- **Reuse:** two-pass engine (`engine.ts`), `ingest.ts` orchestration + `seedServiceOfferings`, `DiscoveryGlimpse` reveal, `compareEnteredVsSite` (built, unwired), `populateCatalog` (built, CLI-only), the Cultivar wizard shell, Ignition's TEAM_QR/progressive patterns as a donor.
- **Net-new:** address extraction in the schema + compare field; the Google/address validation branch + geocoder key; a prospect pre-auth invite token + return route; a background/deferred scrape job; the pre-auth→`owner_id` claim/merge; the structured-query wrapper (thin).

## PART C — THE LANDMINE (decisive answer)

**Question (open since June 8):** does promoting the multi-screen progressive entry + the async invite flow FORCE the identity/auth reconciliation, or can the UI/flow promote clean with identity as a separate footer?

**VERDICT: It cannot promote clean. The entry flow is structurally fused to the auth principal — identity is the SPINE of today's entry flow, not a footer on it. The full async arc forces touching the auth/account-creation ordering (slow, owner-decision). The synchronous reveal + its bug-fixes (Part D, option 1) can promote without touching auth.**

The coupling, in the load-bearing lines:

- **C1 — the entry flow IS the signup flow.** `SignUp.tsx:3-5` imports `OwnerSignup` and registers `DiscoveryGlimpse` as `verticalSteps: [discoveryStep]` (`SignUp.tsx:43`) — a step that renders **after** account creation. `OwnerSignup.tsx:2` imports the live `supabase` client and calls `supabase.auth.signUp(...)` directly (`OwnerSignup.tsx:397`). Discovery is **not** independent of account creation: `DiscoveryGlimpse.tsx:46-48` does `.from('businesses').select('website').eq('id', businessId)` — and that `businessId` exists only because `createBusinessAndMember()` ran after `signUp` (`OwnerSignup.tsx:443,450`).
- **C2 — the invite creates the auth user.** `acceptInvitation.ts` header (`:7`) — "The accepting user has no session yet — they don't exist in auth.users." — then `auth.admin.createUser(...)` (`:85-93`) is the core of accept; `password` is required (`:36-41`). An async invite that stayed **pre-auth** would need a new token-without-account path that does not exist.
- **C3 — the businesses row needs `auth.uid()` first.** `createBusinessAndMember(userId)` stamps `owner_id: userId` as the first column (`OwnerSignup.tsx:254,282`); `userId` is always an existing session, a fresh `signUp().user.id` (`:435`), or `signInWithPassword().user.id` (`:482`). The legacy wizard is identical (`OnboardingWizard.tsx:481-482` bounces to `/login` with no user; insert stamps `owner_id: user.id` `:510-517`). `BusinessProvider` resolution is auth-rooted on both arms (`BusinessProvider.tsx:237,276,310`). **Nothing resolves a business pre-auth.**
- **C4 — "one auth two products" is intent, not wiring.** Both verticals share the `configureAuth` factory (`configureAuth.tsx:42,51`), but sessions are **namespaced per vertical** (`storageKeyFor(vertical)` → `CULTIVAR_OS_DATA` etc., `configureAuth.tsx:38-39`) and live in **two separate Supabase projects** (`bgobkjcopcxusjsetfob` vs `ufsgqckbxdtwviqjjtos`). No cross-product session bridge exists. The builtwithCAI front door does not currently share the vertical's auth principal — it is intended, not built.

**The single line that proves it:** discovery's first query `DiscoveryGlimpse.tsx:48` `.eq('id', businessId)` — and `businessId` is born only at `OwnerSignup.tsx:282`, after `signUp` at `:397`. To run scrape **before** account creation (the whole point of the async invite), you must invert that order, and that inversion **is** the identity/auth reconciliation.

**One nuance that does NOT rescue a clean full-arc promote:** the *capability* layer is auth-free — there is a service-key `/api/discovery/ingest` endpoint and a pre-auth `previewInvitation` (`acceptInvitation.ts:11`). But every **client** entry point that drives them is gated behind a created business/account, and the scraped result is written against a `businessId` that presupposes `owner_id`. The machinery can run sessionless; the flow wiring does not.
