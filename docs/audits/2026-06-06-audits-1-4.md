## Audit 1 — TASK_ROUTING export status

**Finding:** `TASK_ROUTING` in `packages/shared/src/ai/AIEngine.ts` is **NOT exported** — it is a module-internal `const`. Callers outside the file cannot read it without a code change (re-export or accessor). No external consumer currently references `TASK_ROUTING` by name.

**Implication:** Any feature that tries to enumerate or inspect the routing table from outside `AIEngine.ts` will fail silently. Not a current blocker (no caller does this today), but a seam to watch when building the capability registry.

**Verdict:** Known. No change needed today. Flag if a caller ever needs to inspect the routing table — that is the trigger to re-export.

## Audit 2 — AIEngine routing, invoice_scan, VIN OCR, and Gemini vision

### 2a — AIEngine is DARK in Ignition production

**Finding:** `AIEngine.ts` routes all 15 tasks to Railway via `VITE_API_URL`. `VITE_API_URL` is **NOT set** in the ignition-os Vercel project environment. Result: every `AIEngine.call()` in Ignition OS production returns `{ ok: false }`. Ignition AI features are effectively dark in production — they fail gracefully but produce no output.

**Implication:** Any Ignition module that surfaces an AI result (`IgnitionAudit`, `IgnitionCipher`, `PredictiveKey`, `SavingsReport`) has never produced real AI output for an Ignition web user. The `ok: false` graceful path is the only path that has ever run in Ignition Vercel production.

**Railway status:** Railway is load-bearing only for already-broken things → safe to kill clean. Receipt Keeper is Vercel-native from birth, adding no Railway debt. Killing Railway is a cleanup, not a risk.

**Verdict:** Record as TECH-DEBT (existing #12 — expand). Agreed kill path: retire Railway, port only the tasks that have real callers. Orphaned tasks (`invoice_scan`, `vin_decode`) — retire, do NOT port.

### 2b — invoice_scan has ZERO callers

**Finding:** `invoice_scan` appears as a routing entry in `AIEngine.ts` and in `ai_router.py`. It has **ZERO callers** anywhere in the codebase — no component, no hook, no API handler, no page imports or invokes it. It is an orphaned routing entry that was never wired.

**What IS built:** `invoice_audit` — a **different capability** (two-stage Gemini OCR → Claude Sonnet pipeline on Railway). `invoice_audit` reads **outgoing invoices** to flag uncaptured charges (a **leakage** tool — reads what TRACE sends to customers). This is architecturally opposite to Receipt Keeper, which reads **incoming vendor receipts**.

**Verdict:** `invoice_scan` is orphaned — retire it, do not port it. `invoice_audit` is a leakage tool — keep it in scope as a future Ignition capability (post-Railway-kill, port to Vercel). **Receipt Keeper is NET-NEW** — not a promotion of `invoice_scan` or `invoice_audit`. Promoting either would drag Railway architecture in. Promotable from `invoice_audit`: `compressImage()` (image resize utility — trivial, standalone) + the two-stage OCR UI shell pattern.

### 2c — VIN OCR is a placeholder

**Finding:** The VIN decode feature in `packages/ignition-os/modules/IgnitionVIN.jsx` (web stub) and the original mobile implementation both reduce to:
```js
alert('OCR Scanning initializing...')
```
No Gemini vision call is ever made. The feature is a shell that announces it will scan but does nothing.

**The Gemini vision pipeline has never been proven end-to-end on Vercel.** `invoice_audit` is the only real vision implementation in the codebase and it runs on Railway. There is no confirmed, live Vercel serverless function that (a) receives an image, (b) calls Gemini vision, and (c) returns structured output.

**Verdict:** VIN OCR is NOT BUILT as a functional feature. The shell exists. The pipeline needs to be proven on Vercel before any vision feature is marked complete. Gemini Flash OCR for Receipt Keeper will be the first proven Vercel vision pipeline.

## Audit 3 — Storage (receipts / expenses / cost_profile)

**Finding:** **Nothing exists.** Searching the codebase and Supabase schema:
- No `receipts` table
- No `expenses` table
- No `cost_profile` table (nor equivalent)
- One storage bucket exists: `eval-photos` (used by `invoice_audit` image upload in Ignition — not for receipt storage)

No migration file creates any of these tables. No API handler reads or writes to them. No React component references them.

**Proposed AC-clean schema (read-only, proposal — no migration written):**
All tables anchored to `business_id` (AC-1 ✅, AC-2 via `business_id`-scoped RLS ✅):

```sql
-- receipts: one row per vendor receipt captured
receipts (
  id uuid PRIMARY KEY,
  business_id uuid REFERENCES businesses(id),
  image_url text,           -- Supabase storage ref
  vendor text,
  amount numeric,
  category text,            -- from QB Chart of Accounts
  line_items jsonb,         -- OCR-extracted + owner-confirmed
  captured_at timestamptz,
  owner_confirmed boolean DEFAULT false,
  created_at timestamptz DEFAULT now()
)

-- expenses: structured expense records (may be sourced from receipts OR from bank CSV)
expenses (
  id uuid PRIMARY KEY,
  business_id uuid REFERENCES businesses(id),
  receipt_id uuid REFERENCES receipts(id),  -- nullable if CSV-sourced
  source text CHECK (source IN ('receipt', 'bank_csv', 'manual')),
  amount numeric,
  category text,
  occurred_at date,
  created_at timestamptz DEFAULT now()
)

-- cost_profile: owner-declared allocation inputs (intake interview answers)
cost_profile (
  id uuid PRIMARY KEY,
  business_id uuid REFERENCES businesses(id),
  home_office_pct numeric,       -- % of home used for business
  business_time_pct numeric,     -- % of owner's time on this business
  labor_rate numeric,            -- $/hr for owner's own labor
  fixed_overheads jsonb,         -- rent, utilities, insurance (keyed by category)
  asset_amortization jsonb,      -- equipment → { cost, useful_life_yrs, units_produced }
  updated_at timestamptz DEFAULT now()
)
```

**Verdict:** Build from scratch. This is a NEW schema area. The above shapes are a starting point — confirm with the Cost-to-Produce tile spec before writing migrations. The `eval-photos` bucket is Ignition-specific; Receipt Keeper gets its own bucket (`receipts`).

## Audit 4 — Margin engine (shared stub vs. full Ignition engine)

### 4a — Shared marginEngine.ts is a STUB

**Finding:** `packages/shared/src/business-logic/marginEngine.ts` (or `pricing/marginEngine.ts`) contains approximately **17 lines**. It has no:
- Tier definitions (FLEET / LEGACY / FF)
- Overhead calculation
- `analyzeTransaction()` function
- Leakage detection logic
- Price slab breakdown

It is a stub that compiles and exports without error, but produces no real margin analysis. Any importer that calls it in production receives a shell response.

**Current importers of the shared stub:**
- `packages/shared/src/components/SavingsReport.jsx` — calls the stub, receives empty/default output silently

### 4b — Full engine is Ignition-local MarginEngine.js

**Finding:** The complete margin engine is `packages/ignition-os/DataBridge.js` or a co-located `MarginEngine.js`. It implements:
- **4 price slabs** with real tier logic
- **FLEET / LEGACY / FF** customer tiers with per-tier markup overrides
- `analyzeTransaction()` — compares actual price to the tier's calculated price, returns a margin delta
- **Leakage detection** — flags transactions below minimum margin threshold
- `calculateRetail(tx)` — applies markup to `tx.cost` and returns the retail price

### 4c — Overhead is CAPTURED but ORPHANED

**Finding:** `IgnitionProt` (the Margins module in Ignition OS) captures overhead inputs:
- Rent / mortgage
- Electric / utilities
- Fuel
- Maintenance

These are stored in DataBridge `shop_policy.prot_matrix`. However, `calculateRetail(tx)` in the margin engine **ignores them entirely** — it marks up `tx.cost` without adding any overhead-per-unit allocation. The overhead data exists; the engine just doesn't use it.

**Implication:** Every margin calculation in Ignition OS today underestimates the true cost of service. The overhead was captured honestly; the math just doesn't carry it forward.

### 4d — Verdict and build path

**DO NOT promote the shared stub as the base.** The stub has the wrong shape. Port the **full Ignition-local MarginEngine.js** to shared, replacing the stub.

**Wire the orphaned overhead in the same session.** Adding overhead-per-unit to `calculateRetail()` is a near-one-liner once the full engine is in shared: `overhead_per_unit = monthly_overhead / units_per_month`. The data already exists in `prot_matrix`. This is a data-path change, not a rework.

**Cost-to-Produce integration:** Cost-to-Produce feeds **loaded cost** into the existing `tx.cost` slot. The margin engine then marks up that loaded cost. This makes the markup honest (today it marks up purchase cost only). This is a **data-path change**, not a rework of the margin engine itself. The engine already has the slot; it just receives a richer number.

**Full build sequence (agreed):**
1. Port full `MarginEngine.js` → `packages/shared/src/business-logic/MarginEngine.ts` (TypeScript, replace stub)
2. Wire overhead into `calculateRetail()` in the same session (add one overhead-per-unit term)
3. Cost-to-Produce feeds loaded cost into `tx.cost` (data-path change, future step)
